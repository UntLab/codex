# Операционная модель Codex

## Роль `codex`

- `codex` — это не свалка копий, а центральная рабочая база для меня как основного кодера.
- Здесь лежат правила работы, карта проектов, backup-копии, общие skills, автоматизации и внутренние утилиты.
- Отдельные продуктовые репозитории остаются основным местом разработки и источником правды.

## Как начинать задачу

1. Определить проект через `docs/inventory/PROJECT_MAP.md`.
2. Проверить, есть ли у проекта отдельный GitHub-репозиторий.
3. Если отдельный репозиторий есть, по умолчанию работать там.
4. Если проект лежит только в `codex`, работать внутри `codex`.
5. Если нужен только контекст, история или резервная сверка, можно читать `backups/projects/<name>/`.

## Правило source of truth

- `cardsaas`, `bitvantage-yard-console`, `cursor`, `cursor-backups`, `UntLab`, `documentation` считаются отдельными репозиториями со своим source of truth.
- `backups/projects/` не использовать как основное место разработки без прямого запроса пользователя.
- Если пользователь говорит "делай backup" или "сохрани копию в codex", backup-копия обновляется осознанно и отдельно от основной разработки.

## Новый проект

1. Сначала допустима быстрая проработка в `workspace/inbox/`, `workspace/drafts/` или `workspace/experiments/`.
2. Как только проект становится реальным продуктом, предпочтительно создать отдельный GitHub-репозиторий `UntLab/<project-name>`.
3. Разработка дальше идёт в отдельном репозитории.
4. Если нужен резерв в `codex`, копия хранится в `backups/projects/<project-name>/`.
5. Backup-коммиты лучше маркировать явно, например `backup: <project-name> @ <commit>`.

## Когда обновлять карту

- После появления нового репозитория.
- После появления нового backup-проекта в `backups/projects/`.
- После заметного изменения роли проекта или его стека.
- После выделения эксперимента из `workspace/` в отдельный продукт.

## Что читать первым делом

- `AGENTS.md` — локальные инструкции для этой рабочей базы.
- `docs/inventory/PROJECT_MAP.md` — карта репозиториев, backup-копий и ключевых направлений.
- `docs/setup/BACKUP.md` — безопасный sync-режим для самого `codex`.

## Рабочий стандарт для меня

- Держать `codex` как карту памяти и центральную точку входа.
- Не путать backup-копии с активной разработкой.
- По возможности поддерживать короткие, но точные профили проектов, чтобы следующий заход начинался не с разведки, а с работы.
